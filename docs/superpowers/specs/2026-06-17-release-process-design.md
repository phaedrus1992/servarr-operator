# Release Process Design — cargo-release + Keep a Changelog

**Date:** 2026-06-17
**Status:** Approved, pending implementation plan

## Goal

Adopt llmenv's release front-end (Keep a Changelog + `cargo-release`) on top of
servarr-operator's existing publish pipeline. A release must:

- Be driven by a hand-curated, Keep a Changelog `CHANGELOG.md`.
- Use `cargo release <level>` to bump a single shared version, roll the
  changelog, and bump both Helm chart versions in one commit.
- Publish (already working) multi-arch container image + both Helm charts to
  GitHub Packages (GHCR) on a `v*` tag.
- Produce a GitHub Release whose body contains: the curated changelog section,
  Helm install/upgrade instructions, the image reference, and — at the very
  end — the auto-generated `--generate-notes`-style commit/PR list.

## What already exists (do not rebuild)

- `.github/workflows/release.yaml` — triggers on `push: tags: ["v*"]`, computes
  version from the tag, calls `_ci.yaml` then `_publish.yaml`.
- `.github/workflows/_ci.yaml` — clippy + coverage, and a matrix that builds 4
  release binaries uploaded as artifacts `operator-binary-{linux,darwin}-{amd64,arm64}`
  (targets: `x86_64/aarch64-unknown-linux-musl`, `x86_64/aarch64-apple-darwin`).
- `.github/workflows/_publish.yaml` — downloads the 4 binaries, builds & pushes a
  multi-arch (`linux/amd64,linux/arm64`) image to `ghcr.io/phaedrus1992/servarr-operator`,
  packages & pushes `servarr-crds` and `servarr-operator` charts to
  `oci://ghcr.io/phaedrus1992/servarr`, and creates a GitHub Release
  (currently with `--generate-notes`).

## Decisions (resolved during brainstorming)

1. **No crates.io publish.** servarr-operator is an application + internal
   crates, not published libraries. `publish = false`. Release artifacts remain
   image + charts + GH release binaries.
2. **PR-prep release flow** (main is protected). `cargo release` makes the
   prep commit on a `release/*` branch; it does NOT tag or push. The `v*` tag is
   cut on the merged commit, which fires `release.yaml`.
3. **Keep `--generate-notes` content**, embedded at the very end of the release
   body, below the curated changelog + helm instructions.

## Design

### 1. Versioning model — single shared version

- Add `version = "0.1.0"` to `[workspace.package]` in root `Cargo.toml`.
- Change all 4 crates (`servarr-api`, `servarr-crds`, `servarr-resources`,
  `servarr-operator`) from `version = "0.1.0"` to `version.workspace = true`.
- Fix `[workspace.package] repository`:
  `https://github.com/servarr/servarr-operator` →
  `https://github.com/phaedrus1992/servarr-operator`.
- Both Helm charts track the same version (bumped by `cargo-release`, below).

### 2. `CHANGELOG.md` (new, repo root)

Keep a Changelog skeleton with the cargo-release marker comments. Initial
content:

```markdown
# Changelog

All notable changes to this project will be documented in this file.

The format is based on [Keep a Changelog](https://keepachangelog.com/),
and this project adheres to [Semantic Versioning](https://semver.org/).

<!-- next-header -->

## [Unreleased] - ReleaseDate

### Added

- Release automation: `cargo-release` + Keep a Changelog, multi-arch container
  and Helm charts published to GHCR on tag.

<!-- next-url -->
[Unreleased]: https://github.com/phaedrus1992/servarr-operator/compare/v0.1.0...HEAD
```

(Assumes `v0.1.0` is or will be the first tag. If no prior tag exists, the first
release's compare link is regenerated by the rollover rules below.)

### 3. `release.toml` (new, repo root)

```toml
# cargo-release configuration.
# Releases are TAG-TRIGGERED: pushing a `v*` tag fires
# .github/workflows/release.yaml, which builds binaries and publishes the
# multi-arch image + Helm charts + GitHub Release.
#
# cargo-release owns only the local prep: bump the shared version, roll the
# CHANGELOG, and bump both Chart.yaml versions, in one commit. It does NOT
# publish, tag, or push:
#   - publish: there is no crates.io publish for this repo.
#   - tag/push: `main` is protected; the prep commit lands via PR and the `v*`
#     tag is cut on the merged commit. See RELEASING.md.

allow-branch = ["main", "release/*"]
publish = false
tag = false
push = false
shared-version = true
tag-name = "v{{version}}"
pre-release-commit-message = "chore(release): {{version}}"

pre-release-replacements = [
  # Keep a Changelog rollover (cargo-release FAQ pattern).
  { file = "CHANGELOG.md", search = "Unreleased", replace = "{{version}}", min = 1 },
  { file = "CHANGELOG.md", search = "\\.\\.\\.HEAD", replace = "...v{{version}}", exactly = 1 },
  { file = "CHANGELOG.md", search = "ReleaseDate", replace = "{{date}}", exactly = 1 },
  { file = "CHANGELOG.md", search = "<!-- next-header -->", replace = "<!-- next-header -->\n\n## [Unreleased] - ReleaseDate", exactly = 1 },
  { file = "CHANGELOG.md", search = "<!-- next-url -->", replace = "<!-- next-url -->\n[Unreleased]: https://github.com/phaedrus1992/servarr-operator/compare/v{{version}}...HEAD", exactly = 1 },

  # Keep both Helm charts honest in-repo (CI still passes versions at package time).
  { file = "charts/servarr-crds/Chart.yaml",     search = "^version: .*$",    replace = "version: {{version}}", exactly = 1 },
  { file = "charts/servarr-crds/Chart.yaml",     search = "^appVersion: .*$", replace = "appVersion: \"{{version}}\"", exactly = 1 },
  { file = "charts/servarr-operator/Chart.yaml", search = "^version: .*$",    replace = "version: {{version}}", exactly = 1 },
  { file = "charts/servarr-operator/Chart.yaml", search = "^appVersion: .*$", replace = "appVersion: \"{{version}}\"", exactly = 1 },
]
```

Note: the Chart.yaml replacements use anchored `^...$` and rely on
cargo-release's per-file regex (multiline). Validate during implementation that
each matches exactly once; adjust `min`/`exactly` if the templates change.

### 4. Release notes (edit `_publish.yaml`)

Replace the `Create GitHub Release` step. New behaviour builds `release-body.md`
in this exact order:

1. Curated changelog section for this version, via `scripts/changelog-extract.sh <version>`.
2. `## Installing / Upgrading` — Helm instructions (CRDs chart first, then
   operator chart) from `oci://ghcr.io/phaedrus1992/servarr`.
3. `## Container image` — the image reference + tag.
4. `## Full Changelog` (at the very end) — auto-generated notes fetched via
   `gh api repos/{owner}/{repo}/releases/generate-notes` (POST `tag_name`,
   `previous_tag_name`), `.body` appended verbatim.

Then `gh release create "$TAG" --notes-file release-body.md <4 binaries>`,
idempotent (edit + clobber-upload if the release already exists, mirroring
llmenv). `--generate-notes` is NOT passed to `create` — its content is already
embedded at the bottom.

`previous_tag_name` = the most recent `v*` tag before this one
(`git describe --tags --abbrev=0 --match 'v*' "${TAG}^"`, empty on first release).

#### Helm instructions block (rendered into release body)

```bash
# 1. Install/upgrade CRDs + webhook (do this first)
helm upgrade --install servarr-crds \
  oci://ghcr.io/phaedrus1992/servarr/servarr-crds --version <VERSION>

# 2. Install/upgrade the operator
helm upgrade --install servarr-operator \
  oci://ghcr.io/phaedrus1992/servarr/servarr-operator --version <VERSION>
```

`<VERSION>` is substituted with `inputs.chart-version`.

### 5. `scripts/changelog-extract.sh` (new)

`set -euo pipefail`. Takes a version arg; prints the body of the matching
`## [<version>]` section from `CHANGELOG.md` (everything until the next
`## [` header or `<!-- next-url -->`). awk-based, no deps. If the section is
missing, exit non-zero (so the safety rail in §6 can catch it).

Includes a `--self-check` / `main`-guard demo that asserts extraction works on a
small fixture (per repo testing convention).

### 6. Safety rail (edit `release.yaml` `version` job)

Add a step that fails fast when the tag, the workspace version, and the
changelog disagree:

- `tag` (without `v`) must equal `[workspace.package] version` in `Cargo.toml`.
- `CHANGELOG.md` must contain a `## [<semver>]` section (reuse
  `scripts/changelog-extract.sh <semver>` and check non-empty).

This prevents tagging a commit whose changelog/version wasn't rolled.

### 7. `RELEASING.md` (new, repo root)

Documents the operator-facing flow:

```text
# Releasing

1. Ensure CHANGELOG.md [Unreleased] lists the user-facing changes.
2. Install the tool once:  cargo install cargo-release
3. On a release branch:
     git checkout -b release/x.y.z
     cargo release <patch|minor|major>     # bumps version, rolls changelog,
                                            # bumps both charts, ONE commit
4. Open a PR, get it merged to main.
5. Tag the merged commit and push:
     git tag vX.Y.Z <merged-sha>
     git push origin vX.Y.Z
   -> release.yaml builds binaries, publishes image + charts to GHCR, and
      creates the GitHub Release.
```

Note `cargo release` does the version/changelog/chart edits but does not tag or
push (see release.toml).

## Out of scope (YAGNI)

- crates.io publishing and a publish-crate CI job.
- The llmenv website/docs changelog-sync `pre-release-hook` (no docs site here).
- Conventional-commits / git-cliff changelog autogeneration — entries are
  written by hand, Keep a Changelog style.

## Verification

- `cargo release patch --dry-run` shows the intended version/changelog/chart
  edits with no errors and no "Unrendered {{version}}" warning.
- `shellcheck`/`shfmt` clean on `scripts/changelog-extract.sh`.
- `actionlint` + `zizmor` clean on edited workflows.
- `scripts/changelog-extract.sh` self-check passes.
- A dry-run / manual render of `release-body.md` shows the four sections in the
  required order, generated notes last.
